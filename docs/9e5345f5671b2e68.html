Summary

Stephen C. Phillips provides a detailed technical guide for setting up a transparent HTTPS proxy to intercept encrypted traffic from devices on a home network. The primary motivation is reverse-engineering the Tado smart thermostat mobile app's API calls, as Tado does not publicly document its API. The post explains that HTTPS interception requires executing a man-in-the-middle (MITM) attack—which is feasible when targeting one's own network and devices.

The core approach involves three main components: (1) configuring a DD-WRT-enabled wireless router (a D-Link DIR-615) as an access point on a separate subnet, (2) using iptables firewall rules to transparently redirect HTTP and HTTPS traffic to a laptop running Burp Suite proxy, and (3) configuring Burp Suite with "invisible proxying" enabled and installing its CA certificate on the target device (smartphone) to allow decryption of HTTPS traffic.

Phillips methodically walks through router configuration, starting with basic wireless access point setup on subnet 192.168.5.x, then moves to the complex iptables NAT (Network Address Translation) rules needed to redirect traffic. The key iptables commands use PREROUTING (to redirect inbound packets to the proxy), POSTROUTING (to rewrite source addresses so replies return through the router), and FORWARD rules (to allow the proxy traffic). He explains that HTTP uses port 80 and HTTPS uses port 443, so separate proxy listeners on ports 3128 and 3129 are configured. The technical explanation is thorough, including actual output from ifconfig and iptables commands showing which router interfaces (br0 for WiFi, vlan2 for WAN) correspond to different network segments.

Finally, the post covers Burp Suite configuration: creating two proxy listeners bound to "all interfaces" on ports 3128 and 3129, enabling "Support invisible proxying" in the request handling tab, and ensuring the Burp CA certificate is installed on the smartphone. Once configured, all traffic from devices on the test WiFi network automatically appears in Burp's Intercept tab for inspection and modification. Phillips also documents failed attempts using the mangle table and iproute2 routing rules, noting that DD-WRT's stripped-down version of the ip command made debugging impossible. This honest discussion of what didn't work adds practical value.

Key Takeaways

  • HTTPS interception requires a man-in-the-middle (MITM) attack: the attacker creates two encrypted connections (one with the client, one with the server), decrypts both ends, and can inspect/modify traffic in the middle. This is feasible for testing your own devices but difficult to execute undetected on others' devices.
  • Transparent proxying differs from standard proxying: normal proxies require the client to be configured with proxy settings (host/port), but transparent (invisible) proxying intercepts traffic at the network level so the client connects normally while the router redirects it to the proxy without the client's knowledge.
  • iptables operates on three critical chains: PREROUTING (alter packets before routing decisions), POSTROUTING (alter packets after routing), and FORWARD (allow packets to pass between interfaces). Correct NAT rules are needed in both PREROUTING and POSTROUTING to both intercept and return traffic through the router.
  • DD-WRT firmware on commodity routers (like the $50 D-Link DIR-615) provides sufficient control via shell access to configure advanced networking. A separate subnet (192.168.5.x) for the test WiFi isolates traffic and allows applying iptables rules selectively without affecting the main home network.
  • Burp Suite's 'Support invisible proxying' setting enables it to accept connections that weren't explicitly configured as proxies by the client, essential for transparent proxy scenarios. The Burp CA certificate must be installed on the target device so it trusts the proxy's HTTPS certificates.
  • HTTP (port 80) and HTTPS (port 443) traffic require separate proxy listeners (ports 3128 and 3129 in this example) because they need different handling—HTTP can be intercepted directly, but HTTPS requires the proxy to perform TLS decryption and re-encryption with its own certificate.
  • iptables rules must exclude the proxy server itself (using -s ! laptop-ip) to prevent infinite loops where the proxy's own outbound connections get redirected back to itself, which would break functionality.
  • The author's practical motivation—reverse-engineering the Tado smart thermostat API—is a legitimate use case that has produced community documentation and open-source libraries (e.g., node-tado-client on npm), enabling integration with home automation platforms.
  • Failed approaches matter: the author documents attempting the iproute2 mangle table method and explains why it didn't work (missing ip rule show command in DD-WRT). This saves readers from pursuing dead ends.
  • Router interface naming (br0 for bridge, vlan1/vlan2 for virtual LANs, eth2 for physical ports) requires understanding your specific hardware via ifconfig output; generic tutorials won't work without adapting interface names to your router.

Around the same time (2017), CISA issued a formal alert warning that HTTPS interception "weakens TLS security" when used by enterprises or security software. However, Cloudflare's research (2017-2019) found that 10-18% of all TLS traffic to Cloudflare was already being intercepted—by firewalls, antivirus software, and corporate proxies. The post doesn't address these security implications directly; it treats transparent proxying as a neutral technical capability. **Relevance to Wider Conversation**: This tutorial represents the "tinkerer's perspective" on IoT security: when vendors don't document their APIs (Tado explicitly states "we haven't published our API"), reverse-engineering via traffic interception becomes the practical path to integration and automation. The post contributed to a shift where community-maintained API documentation became the de facto standard, eventually leading Tado to provide semi-official API definitions (as seen in GitHub repos like tado-openapispec-v2). **Notable Reactions**: The post spawned practical follow-ups in forum discussions, GitHub repositories, and home automation communities (particularly Home Assistant and openHAB integrations). No public criticism or counter-arguments found; the reception was pragmatic—users either found it useful for their own testing or didn't need the technique. **Reliability & Caveats**: The technical content is accurate and well-tested (evidenced by multiple independent implementations following this guide). However, the post doesn't discuss: (1) SSL/TLS pinning in apps (which defeats transparent proxying), (2) certificate validation bypasses (needed if the target app validates the server certificate), (3) legal/ethical implications when applied to others' devices or networks, or (4) modern defenses like HTTPS Everywhere and certificate transparency logs that would detect such interception in production. The guide is specific to 2017-era tools and may need adaptation for current Burp Suite versions and router firmware. The Tado API information is now partially outdated as Tado has evolved v2.0 and v3.0 APIs, though the networking/proxy techniques remain relevant for any HTTPS interception scenario.
